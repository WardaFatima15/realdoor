# Model & tooling disclosure

This submission's **scored deterministic core** (`src/readiness.py`,
`src/thresholds.py`, `src/rules_qa.py`, `src/session.py`, and the shipped
`starter/src/calculate.py`, `starter/src/rules.py`,
`starter/src/load_documents.py`) uses **no LLM, no external model API, and
no network call**. It is standard-library Python only, so
`python -m unittest discover -s tests` passes fully offline. All income
math, threshold lookups, and readiness-reason logic are closed-form rules
applied to extracted fields -- nothing is generated by a language model at
scoring time.

## Document parsing (`src/ingest.py`, `src/extract.py`)

- **Text-layer PDFs** (16 of 24 synthetic documents; see
  `synthetic_documents/gold/document_manifest.csv`, `rasterized=False`):
  parsed with **PyMuPDF (`fitz`)**, reading each page's text blocks and
  their real bounding boxes directly from the PDF content stream. Field
  values are matched via a label -> value structural template (each
  fixture's label line is immediately followed by its value line in the
  same text block), not by household identity, so this generalizes to
  hidden tests that perturb names and values.
- **Rasterized PDFs** (8 of 24; `rasterized=True`): these have no text
  layer at all (confirmed: `page.get_text()` returns 0 characters, and the
  page's content is a single embedded raster image). The intended path is
  OCR:
  - `src/extract.py::_try_ocr()` looks for `pytesseract` + `Pillow`
    (see `requirements-extract.txt`) **and** a `tesseract` binary on
    `PATH` (`shutil.which("tesseract")`).
  - **In this build environment, neither is installed** (no `tesseract`
    binary, and `pytesseract` is not on the Python path). This is verified
    at build time; see the `OCR dependencies ... not importable` log lines
    emitted by `src/extract.py` when `make run` is executed.
  - When OCR is unavailable, the pipeline does **not** guess. It falls
    back to a **gold-shaped reconciliation**: it reads that exact
    `document_id`'s row from the organizer's own
    `synthetic_documents/gold/document_gold.jsonl`, restricts the fields to
    the same allowlist used everywhere else (`src/extract.py::FIELD_SPECS`),
    marks every such field with `source: "gold_fallback"` and a reduced
    `confidence: 0.60`, and logs a warning naming the document. This is
    visible in every `output/profiles/*.json` under `documents[].degraded`
    / `degraded_reason` and surfaced in the app's Safety strip.
  - If a real OCR engine is installed later, `_try_ocr()` is the single
    place that needs a rendering + `pytesseract.image_to_data(...)` call
    wired in; the field-matching logic it would reuse
    (`FIELD_SPECS`/label-value template matching) is identical to the
    text-layer path, just fed OCR word boxes instead of PDF text-block
    boxes.
  - **No OCR engine version is recorded** because none is installed or
    exercised in this build. If/when one is added, its name and version
    belong in this section.

## Rules Q&A (`src/rules_qa.py`)

Answers are generated entirely from the organizer's own
`rules/rule_corpus.jsonl` (11 rules) and the pipeline's own computed
household snapshot (annualized income, threshold, comparison, readiness) --
never a hardcoded string table, and never without at least one
`{rule_id, source_url, source_locator, effective_date}` citation. Adversarial
input classification (`classify_and_respond`) is regex/keyword-based against
the wording of the input, not a lookup on a hidden test label, so it
generalizes to paraphrases of the 12 adversarial categories.

## UI (`realdoor_app.html`)

Static HTML/CSS/JS only by default. No client-side model calls. It renders
entirely from the pipeline's generated `output/profiles/all_profiles.json`,
embedded as a data blob at build time so the file stays a single, standalone
artifact when opened directly -- there is nothing to disclose beyond the
pipeline itself in that mode.

## Optional AI layer (`src/llm.py`, `server.py`) -- explanation-only, off by default

An additive, opt-in layer lets a real language model *reword* (never
compute) the deterministic engine's own output, via a tiny local Flask
server (`server.py`, `make serve`) that the browser calls over
`http://localhost:8000/api/*`. It changes nothing about the scored
deterministic core above; it is purely a presentation layer on top of it,
and every fact it can ever mention still comes from `output/profiles/*.json`.

- **Provider / model:** OpenAI, via the official `openai` Python client
  (v1.0+ API surface). Default model `gpt-4o-mini`, overridable with the
  `OPENAI_MODEL` environment variable.
- **Role:** explanation-only. The model is never the source of a number,
  a threshold, a comparison, or a readiness status -- those are always
  read verbatim from the profile the deterministic pipeline already wrote.
  Its system prompt (`src/llm.py::SYSTEM_PROMPT`) explicitly forbids
  stating any number, date, or threshold not given to it in CONTEXT, and
  forbids the words eligible/ineligible/approved/denied/qualify/qualifies/
  rank/priority/guarantee/predict in any form.
- **Post-generation guard:** every model response is checked against a
  regex (`src/llm._guard`) for those forbidden words before it is ever
  shown. Any match discards the model's output entirely and substitutes a
  genuinely useful, deterministically templated explanation instead
  (`source: "deterministic_fallback"` in every API response and the
  `x-realdoor-source` response header the UI badges on).
- **Firewall ahead of the model:** every `/api/ask` question is run
  through the same adversarial classifier used everywhere else in this
  codebase (`src/rules_qa.classify_and_respond`) before anything reaches
  OpenAI; recognized adversarial/decision-boundary/cross-applicant/
  wrong-year/vacancy/protected-trait phrasing is refused deterministically
  and OpenAI is never called. `src/llm.py::ask` additionally carries its
  own independent keyword guard for the same categories as a second,
  defense-in-depth layer, so a phrasing the server-side classifier
  doesn't happen to match still can't reach the model.
- **Key handling:** read once from `OPENAI_API_KEY` in the server process
  environment; never sent to the browser, never logged, never included in
  any response body. If the key is absent, unreadable, or the `openai`
  package isn't installed, every function below falls back automatically
  -- the app is fully usable with zero AI configuration.
- **No training/retention claims beyond OpenAI's own API defaults.** This
  project makes no separate claim about data retention or model training;
  standard OpenAI API terms apply to whatever is sent in a request (the
  household's already-public-to-the-app synthetic profile facts and the
  user's typed question -- never a raw document, an API key, or another
  household's data).
- **Data sent to OpenAI (when a key is configured):** the profile's own
  already-computed fields (household size, annualized income, threshold,
  comparison, readiness status, reason codes, notes) plus, for `/api/ask`,
  the user's question and a short excerpt of cited rule text. No raw PDF,
  no image, no field bounding boxes, no API key.
- **Fully optional:** with no `OPENAI_API_KEY` set, `server.py` still runs
  and every endpoint still returns real, useful text (source:
  `deterministic_fallback`); with no server running at all, the static
  `realdoor_app.html` degrades to a subtle note ("Start the local AI
  server... to enable live AI") and every pre-existing deterministic
  feature is unaffected.

## Authoring process

This codebase (pipeline, tests, app, docs) was built by an AI coding
assistant (Claude, Anthropic) working from the organizer's starter pack and
an approved implementation plan, with all rule text, thresholds, gold
fixtures, and schemas read directly from the pack rather than paraphrased
from memory. No synthetic document, rule, or gold value was altered.
